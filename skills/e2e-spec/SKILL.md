# Step2: シナリオ仕様化（e2e-spec）

遷移マップを「経路の羅列」ではなく「**観測点つきの検証仕様**」へ変える。中間観測点が後の漏れ分析（Step4の失敗分類）に効く。

前提: `e2e/plans/<feature>.md` に「遷移マップ」セクションが存在すること（無ければ先に `e2e-map` を実行）。

## 認証の扱い（前提条件 vs シナリオ）

認証必須アプリでは、ここを取り違えると作成者が迷う。次のルールで固定する。

- **認証は「前提条件」がデフォルト。** ログイン済みであることを各シナリオの `開始状態` に `storageState=<role>` として書く。**ログイン操作（email入力→submit）をシナリオの操作列に書かない。** 実体は Step3 の `auth.setup.ts` が用意する（e2e-codegen 参照）。
- **ログインフロー自体の検証を作るのは form（email+password）認証のときだけ。** ログイン成功（happy）・検証エラー・認証失敗（401）・必要ならロックアウトを「ログイン」シナリオ群としてまとめ、これらは未ログイン開始（`storageState=なし`）で書く。
- **SSO / OTP / 2FA / magic link の場合は、ログインフローのシナリオを作らない。** 外部IdP のログイン画面は自動化ブラウザでは bot 検知で弾かれ、安定した検証にならない（自分のアプリの責務でもない）。認証は前提条件として `storageState` で与えるだけにし（Step3 は CDP 接続でログイン済みセッションを採取／e2e-codegen 参照）、`happy path`/`validation error` クラスは「ログイン」ではなく**ログイン後の対象機能**で1件ずつ起こす。`認証:` 欄が form 以外なら、必須網羅クラスの「ログイン成功/検証エラー/401」は **「該当なし（SSO のためログインフローは検証対象外）」と明記**する。なお未ログインで保護URLへアクセスしたときの**ログイン画面へのリダイレクトは permission差分として1件持つ**（これは IdP ログインの自動化ではないので SSO でも書ける）。
- **permission差分クラスは role ごとの storageState に対応づける。** 「一般では非表示／管理者では表示」を、`開始状態: storageState=user` と `storageState=admin` の対比として表現する。未ログインで弾かれる導線（ログイン画面へのリダイレクト等）も permission差分として1件持つ。

## 破壊的シナリオの扱い（この Step の中心責務）

中心原則: **出発点はそのプロダクトを価値たらしめている中心フロー**。価値を通すために破壊的操作（作成/更新/削除/送信など）が必要なら、それはシナリオに含めて当然であり、**破壊的かどうかは経路の属性にすぎない**。破壊的だから除外、ではない。**除外は対等な既定ではなくオプトアウト**で、結果的に破壊的なシナリオがあったときユーザー確認が一度入るだけ。

### 既定は「自己完結シナリオ」

- 価値シナリオは **自己完結（setup → 検証 → teardown）で既定作成**する。自己完結テストは**自分でデータを作り、自分で消す**ので、永続的な汚染を残さない。
- ユーザーが明示的に「これは作らないで」と言ったときだけ除外する。黙って落とさない。
- **「捨て環境かどうか」は本質ではない。** 自己完結テストは作成と削除を自分で完結させるので、向け先（捨て環境か本番類似か）は実行時のユーザー判断に委ねる。シナリオ仕様の段階で捨て環境を必須要件にしない。

### 破壊的シナリオを検出したら「一括提示」する

遷移マップの破壊性属性（Step1 で記録）を見て破壊的シナリオを検出したら、**一括で提示**してユーザーに諮る。1件ずつ逐次質問しない。

- 破壊的シナリオを**表で列挙**し、各行について「**自己完結 / 除外**」を**一度にまとめて決定**してもらう。
- 提示する選択肢は **「自己完結」or「除外」の2択のみ**。「読み取りのみに変える」「skip ガードで眠らせる」は選択肢にしない（価値フローを検証しないテストは負債になる）。
- 捨て環境 path など実行時に必要な情報があれば、その場（一括提示時）でまとめて受け取る。

| 破壊的シナリオ | 破壊性 | teardown 可否 | 既定 | 決定（自己完結/除外） |
|----------------|--------|---------------|------|------------------------|
| タスク作成→完了 | 作成/更新 | 可（UI で削除） | 自己完結 | |
| タスク削除 | 削除 | 可（作成→削除を1本に） | 自己完結 | |
| 招待メール送信 | メール送信 | 不可（外部副作用） | 要慎重 | |

### 無人オーサリング時（ユーザーに確認できない場合）

- **その場でユーザーに確認できない状況で新規に書く**なら、破壊的シナリオは**除外し「要確認」と明示**する。負債になる未検証の破壊的テストを盲目的に書かない。
- 区別は「**人がいる場でのオーサリング vs 無人でのオーサリング**」。一度人と検証済みの破壊的テスト（teardown が確認済み）は、CI で無人実行してよい。無人で禁じるのは「未検証の破壊的テストを新規に書く」ことであって、検証済みテストの実行ではない。

### 不可逆な副作用は人がいても慎重に

- teardown 不能な副作用（**課金・実メール送信・外部通知**など）は、人がいても**除外**するか、**明示的合意**のもとで慎重に扱う。自己完結化できない（消せない）破壊は既定で除外に倒す。

## 必須の網羅クラス（最低1件ずつ）

遷移マップから、以下7クラスを最低1シナリオずつ起こす。対象機能に存在しないクラスは「該当なし（理由）」と明記する。

1. **happy path** — 正常系の主経路
2. **validation error** — 入力検証エラー
3. **permission差分** — 権限による表示/操作の差（未ログイン/一般/管理者など）
4. **戻る操作** — ブラウザ戻る・キャンセルでの状態復帰
5. **再読込** — リロード後の状態保持/喪失
6. **途中離脱** — フォーム入力途中の離脱・再訪
7. **ネットワーク遅延** — 遅延/失敗時のローディング・リトライ・エラー表示

## 各シナリオの必須項目

| 項目 | 内容 |
|------|------|
| 開始状態 | ログイン状態・データ前提・URL |
| 操作 | ユーザー操作の列（番号付き） |
| 中間観測点 | 操作の途中で確認すべき状態（ローディング表示・URL変化・トースト等） |
| 終了条件 | 成功とみなす最終状態（success criteria を曖昧にしない） |
| 後始末（teardown） | **破壊的・自己完結シナリオのみ必須**。作成したデータを何でどう消すか（同じ UI 経路で削除など）。非破壊シナリオは `-`。 |
| 除外事項 | このシナリオで検証しないこと |

## 成果物

`e2e/plans/<feature>.md` に「シナリオ仕様」セクションを追記する。

```markdown
## シナリオ仕様

### S1. ログイン成功（happy path）
- **開始状態**: 未ログイン / 有効な test ユーザーが存在 / `/login`
- **操作**:
  1. email・password を入力
  2. 「ログイン」を押下
- **中間観測点**:
  - 押下後にローディングインジケータが表示される
  - 認証API完了後 `/dashboard` へ遷移
- **終了条件**: ダッシュボード見出しが表示され、ユーザー名が表示される
- **除外事項**: パスワードリセット導線は対象外

### S2. 必須項目未入力（validation error）
- **開始状態**: 未ログイン / `/login`
- **操作**: 空のまま「ログイン」を押下
- **中間観測点**: API は呼ばれない
- **終了条件**: 「メールアドレスを入力してください」等の検証メッセージが表示
- **除外事項**: -

（S3 permission差分 / S4 戻る / S5 再読込 / S6 途中離脱 / S7 ネットワーク遅延 を同形式で続ける）
```

## BDD が必要なチーム向け（任意）

BDD文化のチームは、各シナリオを Gherkin `.feature` にも転記できる（Given=開始状態, When=操作, Then=終了条件, 中間観測点は And で挟む）。ただし二重管理になるため、**Markdown plan を正本**とし `.feature` は派生物として扱う。

完了したら全シナリオを一覧で要約し、**破壊的シナリオの一括提示で確定した「自己完結/除外」の決定**も添えて、**承認ゲート①**（オーケストレーター側）でユーザーのレビューを促す。コード生成（Step3）は承認後に行う。Step3 はここで確定した方針（自己完結なら teardown 込み、除外ならその旨）に従うだけで、勝手に skip ガードで黙らせない。
