# Step1: 到達範囲の地図化（e2e-map）

対象機能の「ユーザー可視の重要フロー」を、コードに落とす前に遷移表として可視化する。目的は**抜けを見える化すること**。いきなりテストケースを書かない。

## まず既存の横断スナップショットを読む（重複回避・未検証優先）

地図化を始める前に、**スイート全体の現況**を把握して、既に他 feature で検証済みの経路を二重に作らないようにする。

1. **`e2e/index.md` があれば最初に読む。** これは `/e2e-audit` が `plans/ tests/ reports/` から毎回再生成する派生スナップショット（横断 coverage 台帳）で、feature 一覧・既検証の class/role・未充足の gap が載っている。手書き台帳ではないので**読むだけ**（ここでは書き換えない。出力は Step4 後の `/e2e-audit` が行う）。
2. **`e2e/index.md` が無ければ `e2e/plans/*.md` を直接スキャン**して、既存 plan のシナリオ（class/role/`coverage` メタ）を素材にする。
3. **重複経路は `covered_elsewhere` 候補として扱う。** 今回の対象機能の経路が、既存 feature の plan で既に検証済みなら、その経路は本 feature で作り直さず Step2（e2e-spec）で `status=covered_elsewhere` にする候補として申し送る（どの feature で検証済みかを併記）。
4. **未検証・gap の経路を優先する。** index.md（または plans スキャン）が示す「まだ誰も踏んでいない class/role/経路」を、今回の地図化で優先的に埋める。

> どちらも無い（初回・他 plan が存在しない）場合はこの手順を飛ばし、通常どおり価値フロー起点で地図化する。

## 出発点は「価値を生む中心フロー」

地図化は**画面の棚卸し（全画面を機械的に列挙）ではなく、そのプロダクトを価値たらしめている中心フロー**から立てる。「ユーザーがこのプロダクトで達成したい中心的な成果は何か」を起点に、その成果に至る経路と分岐を辿る。価値フローを辿った結果、作成・更新・削除・送信などの**破壊的な操作が経路に含まれるのは自然なこと**であり、それを理由に経路を地図から落とさない。破壊性は経路の**属性として記録するだけ**で、扱いの判断（自己完結化するか除外するか）は Step2（e2e-spec）の責務。

> **破壊性の定義**: 永続的な状態変更（作成/更新/削除/送信）＋外部副作用（課金/メール/通知）を伴う操作。遷移表の「破壊性」列に記録する。「破壊的だから探索やシナリオ化を避ける」とは判断しない（除外は対等な既定ではなく、Step2 でのユーザーのオプトアウト）。

### 価値フローはコードからは演繹できない — 不明なら「聞いて確定する」

**コードは「how（どんな画面・経路・状態があるか）」は教えるが、「why／何がこのプロダクトの価値か」は教えない。** 画面構造・ルート・権限などの**構造的事実はコード/seed/docから取れるだけ取る**。だが「どのフローが中心的価値か」は別種の情報で、ドキュメント化されていなければ**推測で決め打ちせず、ユーザーに確定させる**。「推測で画面を捏造しない」のと同じく、**推測で価値フローを決め打ちしない**。

次の段階で確定する（**前段で取れたものは飛ばす**）:

1. **コード / seed test**: ルート定義・コンポーネント・既存テストから、存在する画面・経路・ロールを把握する（構造の素材）。
2. **PRD / 仕様 doc**: 「何のための機能か・誰がどんな成果を得るか」が書かれていれば、そこから中心フローを起こす。無ければユーザーに**ドキュメントの提示を求める**。
3. **それでも中心価値が不明なら、ユーザーに質問する**（下記）。質問は**まとめて一括**で投げる（1問ずつ往復しない）。回答が得られるまで遷移マップの中心フローを確定しない。

価値を確定する質問の型（対象に合わせて3〜5問・例）:

- このプロダクト／機能で、ユーザーが達成したい**中心的な成果**は何か？（1文で）
- その成果に至る「**これが通れば価値が成立する**」と言える代表フローは？（複数あれば優先順位も）
- そのフローに**欠かせない前提**（ロール・初期データ・状態）は？
- 逆に「**今回は対象外**」と言える範囲は？（スコープの線引き）
- **成功**と判断する最終状態は、どの画面・表示・通知で確認できるか？

> 価値の所在さえ固まれば、画面の網羅や分岐の洗い出しはコード/探索で機械的に埋められる。**曖昧にしたまま探索に入って「画面の棚卸し」に堕ちるのを防ぐのがこの確定作業の目的。** ここで詰め切れない論点は「未確認・要レビュー」に残し、承認ゲート①へ送る。

## 入力源（あるものを使う・優先順位付き）

1. **PRD/仕様ドキュメント**: ユーザーが指定した仕様書・README・チケットを読み、画面・遷移・分岐・前提を抽出する。
2. **実サイト探索**: 起動中のアプリがあれば、**chrome-devtools MCP を優先**して探索する（動作が軽い）。`take_snapshot` で ARIA 相当の構造を取り、`navigate_page` / `click` で遷移を辿る。chrome-devtools MCP が使えない場合のみ playwright MCP にフォールバック。
3. **既存 seed test / コード**: 既存の Playwright test やルート定義・コンポーネントを文脈に入れ、認証状態・既知の画面構造を把握する。

どれも無い場合は、ユーザーに「対象URL・PRD・seed のいずれかを提示してほしい」と求める。**推測で画面を捏造しない。**

## 認証必須アプリの場合（最初に確定させる）

ログイン後の画面は、未ログインのブラウザでは探索できない。探索を始める前に次の3点を確定し、ヘッダーの `認証:` 欄を埋める。**ここを曖昧にしたまま進むと、後段で storageState 戦略が崩れて作成者が詰まる。**

1. **ログイン手段を「利用者が観測できる言葉」で確定する** — 必須の問いはこの二択だけ:「ログインは**メールアドレス＋パスワードを画面に入力する形式**か？」
   - **はい** → `E2E_AUTH_MODE=form`。`auth.setup.ts` が env の資格情報で自動ログインし、storageState を自動採取する。
   - **いいえ**（SSO／外部IdP・OTP・2FA・magic link・生体認証 等）→ `E2E_AUTH_MODE=prebuilt-state`。利用者が手元で CDP スクリプト（`save-state-cdp.ts`）により手動採取してからテストを作成する（README のフォールバック手順）。**予告: この方式では後で手元での storageState 採取（実 Chrome 起動＋手動ログイン）が必要になる。具体手順は Step3（収束ループ入口）で案内する**（form/none では不要）。
   - **保存場所（cookie / localStorage / IndexedDB）は問わない。** 採取は3種すべてを自動でカバーするため、ここで宣言させない。Firebase 等トークンを **IndexedDB** に置くアプリも form 自動採取で対応される（Playwright の `storageState({ indexedDB: true })`、1.51+）。
   - form を選んでも、自動ログインが **bot 検知 / captcha** で弾かれたら `prebuilt-state`（手動採取）へフォールバックする。
2. **ロール一覧を確定する** — `未ログイン` / `一般ユーザー` / `管理者` など。ここで挙げたロールが Step2 の **permission差分クラス**の素地になる。
3. **探索手段は「既存ログイン済みセッションに乗る」を優先** — ユーザーに対象アプリへ手動ログインしてもらい、その実セッションに **chrome-devtools MCP** で乗って探索する。未ログイン状態なら「先に手動でログインしてから探索を開始する」とユーザーに促す。探索のためにアプリを自動ログインさせる必要はない。
   - **普段使いの Chrome（既存ログイン済み）に接続する設定**: chrome-devtools MCP を既定で起動すると別の使い捨てプロファイルで新規ブラウザを立ち上げてしまい、ユーザーのログイン状態に乗れない。MCP の起動引数に **`--autoConnect`** を付けると、起動中の Chrome にそのプロファイルごと自動接続でき、実ログインセッションのまま探索できる（`pnpm exec chrome-devtools-mcp@latest --autoConnect`）。
   - **`--autoConnect` で繋がらない / DevToolsActivePort が無い場合のフォールバック**: ユーザーに `--remote-debugging-port=9222` 付きで Chrome を起動してもらい、MCP を **`--browser-url=http://127.0.0.1:9222`** で明示接続する。それも難しければ、**Claude-in-Chrome 拡張**で実ログインセッションのタブに乗って探索する（拡張は既存ブラウザのセッションをそのまま使える）。

## 守るべき原則

- **コードは書かない**（spec生成は Step3 / e2e-codegen の責務）。
- **未確認領域は「未確認」と明記する**。探索できなかった分岐・権限・状態は表に「未確認」と書き残す。
- **探索時の安全策**として、フォーム送信・課金・破壊的操作は実行しない（探索中に実体を壊さないため。観測点として記録するに留める）。これは探索手段の話であり、**破壊的な経路を地図やシナリオ化から落とす意味ではない**（その経路は破壊性属性を付けて表に残す）。
- 主経路だけでなく、**戻る・分岐・エラー経路・非同期ポイント**を意識して列挙する（探索不足が最大の漏れ原因）。

## 成果物

`e2e/plans/<feature>.md` に「遷移マップ」セクションを書く（ファイルが無ければ作成、あれば該当セクションを更新）。

```markdown
# E2E Plan: <feature>

> 生成日: <YYYY-MM-DD> / 入力源: <PRD | 実サイト探索(chrome-devtools) | seed test>
> 対象URL: <baseURL/path> / 認証: <要/不要・方式>
> 探索環境: ロケール=<例: ja-JP> / タイムゾーン=<例: Asia/Tokyo>
>   ↑ 実行環境(playwright.config の use.locale/timezoneId)で必ず同じ値に固定すること。
>     探索時と実行時でズレると、ローカライズ文言・日付の assert が「環境依存」で落ちる。

## 遷移マップ

### 前提データ / 環境
- ログイン状態: <未ログイン | role=...>
- 必要なシードデータ: <...>
- 除外範囲: <外部決済・メール送信実体 など>

### 画面・遷移表

「破壊性」列には、その操作が永続的状態変更＋外部副作用を伴うなら種別（作成/更新/削除/送信/課金/メール/通知）を、伴わない読み取り中心なら `-` を記す。**この列は属性の記録であって除外フラグではない**（自己完結化するか除外するかの判断は Step2）。

| # | 画面 | 到達操作 | 分岐条件 | 非同期イベント | 破壊性 | 期待結果 | 確認状況 |
|---|------|----------|----------|----------------|--------|----------|----------|
| 1 | ログイン画面 | `/login` へ遷移 | - | - | - | フォーム表示 | 確認済 |
| 2 | ダッシュボード | 正しい認証で submit | 認証成功 | 認証API応答待ち | - | 一覧表示 | 確認済 |
| 3 | エラー表示 | 不正な認証で submit | 認証失敗 | 認証API応答待ち | - | エラーメッセージ | 未確認 |
| 4 | タスク作成 | 「新規」→保存 | 必須項目あり | 保存API応答待ち | 作成 | 一覧に追加 | 確認済 |
| 5 | タスク削除 | 行メニュー→削除 | 確認ダイアログ | 削除API応答待ち | 削除 | 一覧から消える | 確認済 |

### 状態グラフ（条件付きフォーム・段階UIがある場合）
- 段階1: <...> → 段階2: <...>（中間状態を飛ばさない）

### 未確認・要レビュー
- <探索できなかった経路・権限差分・エラー系>

### 未確定（Step2 進行判定）

Step2（e2e-spec）に進めるかの**契約**。次の4項目のうち一つでも「未確定」が残るなら、オーケストレーターは Step2 に進まずユーザー確認で止める（`commands/e2e-plan.md` 参照）。確定済みなら「確定」と書き、未確定なら理由を残す。**teardown 可否はここに含めない**（破壊的シナリオの自己完結/除外は Step2 のゲートで諮るため）。

- [ ] **中心価値**: このプロダクトの「価値を生む中心フロー」が1文で言えるか（推測でなくユーザー/PRDで確定したか）
- [ ] **ログイン手段**: 「メールアドレス＋パスワードを画面入力する形式か？」が確定したか（はい=`form` 自動採取 / いいえ=`prebuilt-state` 手動採取 / 認証不要=`none`。保存場所 cookie/localStorage/IndexedDB は問わない）
- [ ] **対象ロール**: 検証対象のロール一覧（未ログイン / 一般 / 管理者 など）が確定したか
- [ ] **画面確認手段**: 実サイト探索・PRD・seed test のいずれかで画面を確認できたか（どれも無く画面未確認なら「未確定」）
```

完了したら、見つかった経路数・「未確認」項目・**破壊性を持つ経路の一覧**を要約してユーザーに伝え、次の `e2e-spec`（Step2）へ渡せる状態であることを示す。破壊的経路の扱い（自己完結化／除外）は Step2 でまとめて判断するため、ここでは判断せず属性として申し送るだけにする。**併せて「未確定（Step2 進行判定）」の4項目の確定/未確定を必ず明示する**——ここに未確定が残るかどうかが、オーケストレーターの Step1→Step2 連続実行を止めるかの判定材料になる。
